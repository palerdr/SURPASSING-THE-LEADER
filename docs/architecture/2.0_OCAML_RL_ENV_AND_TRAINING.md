# OCaml RL Environment And Training Design

This document explains how to wrap the engine for RL and how to store the pieces in a way that stays debuggable.

The previous overview was the strategic sketch. This version is the implementation-facing one.

## Main Goal Of The RL Layer

The engine should answer:

- what the rules do,
- what happened after a joint action,
- whether the game ended.

The environment should answer:

- what the agent can see,
- what action indices mean,
- what is legal right now,
- what reward to emit,
- and who chooses the opposing action.

Keep that separation strict.

## Recommended Environment Loop

The underlying game is simultaneous, so your RL wrapper has to collect both actions before calling the engine.

Recommended step flow:

1. inspect current engine state
2. determine the controlled player's role
3. build observation
4. build legal-action mask
5. decode the agent's chosen action index into a time value
6. ask the opponent policy for its time value
7. package those into a joint action
8. call engine step
9. compute reward from before/after states
10. emit next observation and metadata

## What To Store In `env_state`

Use a record.

Recommended fields:

- `engine_state`
- `config`
- `controlled_player`
- `opponent`
- `last_observation`
- `last_action_mask`
- `done`
- `episode_return`
- `episode_steps`
- `debug_info`

## Opponent Storage Options

### Option A: function value

Store a callable policy-like function.

Pros:

- lightweight,
- simple for scripted bots.

Cons:

- less descriptive if you later want metadata.

### Option B: record with callbacks, recommended

Example conceptual fields:

- `name`
- `policy_kind`
- `act`
- `reset`

This is better because your environment can report which opponent was used.

## Action Space Design

The cleanest action is a chosen second.

## Recommended storage

For RL, use a fixed discrete index space and decode it.

### Best first design

- indices `0..60`
- map them to seconds `1..61`

Then use a mask:

- checker usually only has seconds `1..60`
- dropper has `1..60` normally and `1..61` during a leap turn

## Why fixed-size action space is helpful

- easy PPO implementation,
- one policy can handle both roles,
- action masks remove illegal choices,
- no role-specific output reshaping needed.

## Alternative design: separate heads

Possible but more work.

Use only if you later decide role-conditioned heads improve training stability.

## Action Encoding Helpers

Useful environment-side helper functions:

### `action_index_to_second`

Maps raw action index to a human game second.

### `second_to_action_index`

Useful for scripted policies or imitation data.

### `is_action_legal`

Checks whether a given index is legal in the current role/window.

### `legal_action_mask`

Returns a `bool array` or `float array` mask.

Storage choice:

- use `bool array` in pure OCaml logic,
- convert to float mask only when needed for torch integration.

## Observation Design

Do not jump directly to raw arrays if you want this to stay understandable.

## Recommended approach

1. build a typed observation record
2. flatten it into `float array`
3. convert that into a tensor only at the edge

## Suggested observation record

Fields worth including:

- `self_role`
- `self_cylinder`
- `opp_cylinder`
- `self_ttd`
- `opp_ttd`
- `self_deaths`
- `opp_deaths`
- `referee_cprs`
- `round_index`
- `half_index`
- `turn_duration`
- `leap_turn`
- `self_safe_checks_remaining`
- `opp_safe_checks_remaining`
- `clock_normalized`
- `minutes_until_leap` or similar derived feature
- `history_features` later if needed

## Storage options for observations

### Option A: normalized floats only

Good for ML, bad for human inspection.

### Option B: mixed typed record, recommended

Use ints/bools/variants in the record and normalize only in `to_feature_array`.

This is far easier to debug.

## Feature Normalization Guidance

Normalize values in predictable ways.

Examples:

- cylinder by dividing by max cylinder
- TTD by dividing by max relevant horizon
- CPR count by dividing by a cap or using log scaling
- clock by dividing by a total expected match horizon or using sin/cos style features if you want periodic time encoding later

Keep the normalization logic in one place, not scattered through training code.

## Information Regimes

You probably want two observation modes.

### Full-information mode

The observation directly reflects engine truth.

Good for:

- early engine validation,
- baseline RL,
- quick debugging.

### Knowledge-limited mode

The observation only includes what the player could reasonably infer.

Good for:

- later experiments,
- more faithful strategic AI,
- school-project discussion about imperfect information.

## How To Store Observation Mode

Use a variant in env config, not a loose boolean.

For example conceptually:

- `Full_information`
- `Player_knowledge`
- maybe later `Hal_perception`

This keeps the meaning obvious.

## Reward Design

Start simple.

## Recommended baseline

- win = `+1.0`
- loss = `-1.0`
- nonterminal = `0.0`

## Why this is a good baseline

- faithful to the real objective,
- easy to interpret,
- harder to accidentally train weird incentives.

## If shaping is needed

Add it in a structured way.

Use a reward breakdown record such as:

- `terminal`
- `risk_delta`
- `route_progress`
- `strategic_bonus`
- `total`

This is better than adding magic constants inline.

## Useful shaping candidates

- reward for increasing opponent cylinder risk,
- penalty for increasing your own fatal exposure,
- reward for moving toward favorable leap-second parity,
- tiny penalty for illegal-action attempts if your agent can make them,
- small bonus for causing opponent predictability only if you can justify it.

## Reward functions to consider

### `terminal_reward`

Only checks match end state.

### `risk_delta_reward`

Compares before/after threat levels.

### `route_progress_reward`

Captures whether the game moved toward or away from favorable leap-second alignment.

### `transition_reward`

Combines all chosen reward components.

## Opponent Interface

All opponents should fit the same interface.

Recommended conceptual inputs:

- current engine state or observation
- controlled role
- legal action mask
- optional RNG

Recommended outputs:

- chosen action index or chosen second
- optional debug reason / tag

## Good opponent types

- random timing opponent
- safe-strategy bot
- greedy accumulation bot
- route-aware heuristic bot
- scripted Hal planner
- learned policy bot

## Why a scripted Hal matters before RL

Your final project needs strong strategic behavior that is understandable.

If you implement a good route-aware heuristic Hal first, you get:

- a strong baseline,
- better curriculum for RL,
- imitation targets,
- and a way to compare learned behavior against intended behavior.

## Rollout Storage

If you do PPO, create an explicit rollout record type rather than juggling parallel arrays early on.

Suggested fields per transition:

- observation
- action index
- action logprob
- value estimate
- reward
- done
- action mask
- optional engine summary

Then your rollout batch can just be a list or array of transition records.

## Why store masks in rollouts

Because if you later need to recompute policy losses or debug invalid behavior, it helps to know what was legal at sample time.

## PPO Design Guidance

Recommended first design:

- one shared policy/value backbone,
- one categorical head over 61 logits,
- one value head,
- legal-action masking before sampling and loss,
- role encoded as an input feature.

## Why shared role-conditioned policy is good here

- same game for both roles,
- fewer moving parts,
- easier self-play,
- easier comparison between strategic regimes.

## Training Config Storage

Keep training config separate from engine config.

Suggested categories:

- PPO hyperparameters
- network dimensions
- rollout length
- minibatch settings
- entropy/value coefficients
- opponent schedule
- seed list
- checkpoint frequency
- evaluation settings

Use records and serialize them cleanly.

## Self-Play Storage Strategy

If you add self-play, maintain an opponent pool.

Useful opponent-pool entry fields:

- checkpoint path
- creation step
- evaluation score
- selection weight
- metadata tags

This is better than just a folder full of nameless checkpoints.

## Episode Metadata Worth Returning

Your `step_result` or episode summary should include more than reward.

Useful info:

- whether a death occurred
- death duration
- leap-turn flag
- resulting round and half
- cylinders after step
- match terminal flag
- winner if terminal
- route variation if you later expose it

This is extremely useful for debugging training.

## Reproducibility Requirements

For a school project, reproducibility matters a lot.

Always save:

- engine config
- env config
- reward config
- training config
- random seeds
- exact code revision if using git
- checkpoint metadata
- evaluation opponent list

## First Practical Build Order

1. finish the pure engine
2. build a typed observation record
3. build action decoding and masks
4. build one scripted opponent
5. build the env `reset`/`step`
6. train a tiny PPO against random and safe bots
7. add richer heuristic opponents
8. later add self-play or planner imitation

## Strong Recommendation For Your Project

Because the game is mathematically structured and your final target is Hal rather than a generic policy, I would strongly recommend this sequence:

1. exact engine
2. route-aware scripted Hal
3. env + PPO
4. use RL to improve robustness, not to invent the whole strategy from nothing

That will give you a project that is both impressive and explainable.

# OCaml Engine Module Spec

This file is the concrete module-by-module build guide.

Instead of only listing module names, it explains:

- what each module stores,
- what functions belong there,
- what each function should do,
- what it should return,
- and which design choices are reasonable.

## `Config`

Purpose: store every tunable rule parameter in one place.

## Suggested internal structure

Use a record composed of sub-records.

Example conceptual grouping:

- `timing_config`
- `cylinder_config`
- `survival_config`
- `player_config`
- `rule_flags`

## What should live here

- standard turn duration,
- leap turn duration,
- failed-check penalty,
- cylinder max,
- within-round overhead,
- between-death overhead,
- leap-second window,
- survival constants,
- default physicality values,
- optional switches for alternate rule interpretations.

## Useful functions

### `default`

Returns the project's standard rule set.

### `validate`

Checks for nonsense config values.

Examples:

- leap duration must be >= normal duration,
- cylinder max must be positive,
- penalties must be nonnegative,
- floors must be within `[0, 1]`.

### `for_difficulty`

Optional helper if you later want different presets.

### `with_overrides`

Useful for experiments and tests.

## `Domain`

Purpose: hold shared types with minimal dependencies.

## Recommended contents

- `player_id`
- `role`
- `half_index`
- `life_status`
- `half_result`
- `finish_reason`
- `match_status`
- `validation_error`
- shared summary records

## Storage guidance

Prefer variants for categorical states.

Bad fit for strings:

- roles,
- half identifiers,
- terminal reasons,
- result classes.

These should all be variants.

## `Clock`

Purpose: be the only place that knows how time works.

## Suggested stored types

You can use raw `int` seconds for the actual clock.

If you want extra clarity, define type aliases like:

- `type game_second = int`
- `type turn_duration = int`

These are still ints underneath but document intent.

## Core functions

### `in_leap_window`

Checks whether a half-turn starting at this absolute game second gets leap-second treatment.

Input:

- config
- absolute game second

Output:

- bool

### `turn_duration_for_start_time`

Returns the legal drop window for a half-turn beginning now.

Responsibility:

- return normal duration normally,
- return leap duration in the leap window.

### `advance`

Adds elapsed time to the absolute game clock.

Keep this very dumb. It should be plain arithmetic.

### `snap_to_next_round_boundary`

Moves the clock to the next round-start boundary after a full round.

This is the tricky one.

Document exactly how it behaves:

- pre-leap,
- at the leap second,
- post-leap.

### `to_wall_time`

Converts absolute game seconds into a display record such as:

- hour,
- minute,
- second.

This can be cleaner than formatting directly.

### `format_wall_time`

Builds a UI/CLI string from `to_wall_time`.

## Optional helper functions

- `is_round_boundary`
- `minute_index`
- `seconds_until_next_boundary`

## `Player`

Purpose: encapsulate all local player-state transitions.

## Recommended `player_state` fields

- `id`
- `name`
- `physicality`
- `cylinder_seconds`
- `ttd_seconds`
- `deaths`
- `status`
- `death_history`

## Core functions

### `create`

Construct a fresh player state.

### `add_to_cylinder`

Input:

- player,
- amount.

Output:

- updated player,
- overflow info.

Useful overflow return shapes:

Option A:
- `player * bool`

Option B, better:
- `player * [ `No_overflow | `Overflow ]`

Option C, richest:
- `player * overflow_report`

Where `overflow_report` can include the exact post-update amount.

### `set_cylinder`

Helpful for tests and special scenario setup.

### `record_death`

Should:

- increment death count,
- add duration to TTD,
- append to history,
- mark temporary death.

### `revive`

Should:

- reset cylinder,
- set status alive,
- keep TTD and death count unchanged.

### `mark_permanent_death`

Should:

- mark status permanently dead.

### `safe_checks_remaining`

Should compute how many worst-case safe checks remain from current cylinder.

This function is strategically important, so unit test it heavily.

## Optional helpers

- `can_survive_penalty`
- `remaining_capacity`
- `is_alive`
- `is_terminal`

## `Referee`

Purpose: store the referee's global state.

## Suggested state

- `cprs_performed`
- optional fatigue or condition fields

## Core functions

### `create`

Fresh referee.

### `record_cpr_attempt`

Increment CPR count after any revival attempt.

### `effective_quality`

Optional helper returning a derived scalar if you later enrich the system.

You may keep this module tiny at first and let `Survival` consume its state.

## `Survival`

Purpose: compute revival odds and sample the outcome.

## Suggested output type

Use a record for rich outputs if possible.

For example, conceptually:

- base factor
- cardiac factor
- referee factor
- physicality factor
- final probability

This is excellent for testing and later analysis.

## Core functions

### `base_curve`

Use only the current death duration.

### `cardiac_modifier`

Use only prior accumulated total dead time.

### `referee_modifier`

Use only referee fatigue or CPR count.

### `survival_breakdown`

Recommended helper.

Instead of only returning a float, return the whole factorization.

Why it is useful:

- easy to debug,
- easy to print,
- easy to test against the math in your write-up.

### `survival_probability`

Can simply call `survival_breakdown` and extract final probability.

### `attempt_revival`

Input:

- RNG state
- survival probability

Output:

- survived flag
- updated RNG state
- maybe sampled roll if you want traceability

If you store the sampled roll in history, debugging gets much easier.

## `Rules`

Purpose: resolve the local mechanics of a single half-turn without touching the full game state machine.

## Suggested local outcome record

Recommended fields:

- `drop_time`
- `check_time`
- `turn_duration`
- `success`
- `st_gained`
- `penalty_applied`
- `checker_before`
- `checker_after_local_update`
- `injection_triggered`
- `death_duration`
- `result_class`

This is separate from the final `half_round_record`, which the `Game` module can build after death/revival and clock advancement are known.

## Core functions

### `validate_drop_time`

Checks range against the actual turn duration.

### `validate_check_time`

Checks legal checker action range.

### `validate_joint_action`

Convenience wrapper to validate both at once.

### `check_succeeds`

Pure timing comparison.

### `compute_st`

Pure ST arithmetic for successful checks.

### `resolve_success_case`

Optional helper.

Useful if you want to isolate the successful-check branch:

- add ST,
- detect overflow,
- construct local outcome.

### `resolve_failure_case`

Optional helper.

Useful if you want to isolate the failed-check branch:

- add fixed penalty,
- trigger injection,
- construct local outcome.

### `resolve_half_outcome`

Main local resolver.

Should:

- validate action legality,
- decide success/failure,
- update checker local state,
- decide whether death is triggered,
- return a structured outcome.

It should not:

- sample survival,
- pick a winner,
- advance global clock,
- append game history.

## `Game`

Purpose: the authoritative orchestrator.

## Suggested `game_state` fields

- `player_a`
- `player_b`
- `referee`
- `game_clock`
- `round_index`
- `current_half`
- `first_dropper`
- `match_status`
- `history`
- `rng`

## Suggested return shape for stepping

Better than returning only the new state:

- return the new state plus the produced record,
- or return a step-result record containing both.

That avoids re-reading history just to inspect what happened.

## Core functions

### `init`

Builds a valid initial state.

Decide whether round indexing starts at `0` or `1` and never be inconsistent.

I recommend:

- `round_index = 1` for readability in history.

### `seed`

Optional helper if you store RNG in the state.

### `roles_for_half`

Returns the current dropper and checker.

### `player_by_id`

Useful helper for summaries and UI.

### `replace_player`

Very useful immutable-update helper when you have named player fields.

### `step_half_round`

This is the main match transition function.

Suggested internal order:

1. reject stepping if match is terminal
2. derive roles
3. derive turn duration from clock
4. resolve local rules outcome
5. if death occurred, compute survival probability
6. sample revival
7. update checker and referee states
8. update terminal status if needed
9. advance turn clock
10. advance death overhead if needed
11. update half/round progression
12. build final half-round record
13. append history

### `step_round`

Convenience wrapper for two sequential half-rounds.

### `current_turn_duration`

Convenience wrapper over `Clock.turn_duration_for_start_time`.

### `is_terminal`

Simple status check.

### `winner`

Return winner id if terminal.

### `state_summary`

Return a compact record or map for UI/tests.

## `Trace`

Purpose: analysis and replay utilities.

## Recommended functions

### `history_to_lines`

Human-readable string list for CLI output.

### `to_csv_rows`

For easy export to spreadsheets.

### `round_timeline`

Useful for inspecting route deviations and leap-second alignment.

### `compute_match_stats`

Can summarize:

- total deaths,
- final TTD values,
- max cylinder values,
- whether leap second occurred while active,
- number of safe strategies used.

## `Observation`

Purpose: build RL-facing views of the engine state.

## Observation storage options

Option A: directly produce float arrays
- fastest,
- hard to debug.

Option B: use a typed observation record first, recommended
- easier to reason about,
- easier to unit test,
- convert to arrays only at the final step.

## Suggested observation record contents

- self cylinder
- opp cylinder
- self TTD
- opp TTD
- self deaths
- opp deaths
- referee CPRs
- current role
- current round
- current half
- leap-turn flag
- safe-check budget for both
- optional normalized clock features
- optional recent-action features

## Core functions

### `from_game_state`

Produce a typed observation from one player's perspective.

### `to_feature_array`

Flatten observation into `float array`.

### `legal_action_mask`

Return which action indices are valid for the current perspective and role.

## `Reward`

Purpose: compute training rewards from transitions.

## Reward storage options

Option A: one raw float
- enough for PPO.

Option B: breakdown record plus total, recommended during experimentation
- terminal component
- risk component
- route component
- shaping component
- total

This makes shaping much easier to debug.

## Core functions

### `transition_reward`

Input:

- perspective player id
- pre-state
- post-state

Output:

- total reward or breakdown.

### `terminal_reward`

Clear helper to isolate end-of-match logic.

### `route_progress_reward`

Optional if you later want leap-second-related shaping.

## `Env`

Purpose: present a turn-based RL API around a simultaneous-move engine.

## Suggested `env_state`

- engine state
- controlled player id
- opponent policy handle
- latest observation
- done flag
- cumulative reward
- episode stats

## Core functions

### `reset`

Create a fresh episode.

Should return:

- initial env state,
- initial observation,
- maybe action mask.

### `decode_action`

Map an RL action index to a concrete game timing.

### `query_opponent`

Ask the scripted or learned opponent for its action.

### `step`

Suggested order:

1. decode agent action
2. get opponent action
3. produce joint action
4. step engine
5. build next observation
6. compute reward
7. update episode stats
8. return step result

### `action_mask`

Return valid actions at the current state.

### `episode_stats`

Return useful info at end of episode.

## Testing Priorities By Module

### `Clock`

- leap boundary behavior
- post-leap minute snapping
- display continuity

### `Player`

- cylinder additions
- overflow at threshold
- death bookkeeping
- safe-check counting

### `Survival`

- fresh death probability
- fatal threshold behavior
- cardiac decay
- referee fatigue decay

### `Rules`

- success vs failure
- ST computation
- failed-check penalty
- overflow-triggered injection

### `Game`

- role swapping
- half-to-half progression
- round boundary snapping
- terminal death
- history record correctness

### `Observation` / `Env`

- perspective correctness
- mask correctness
- reset/step consistency

## Best Default Choices If You Want To Start Coding Fast

If you want sensible defaults without overthinking everything:

- use records for all major state,
- use variants for statuses and result classes,
- store players as named fields,
- store history as a list,
- store RNG in `game_state`,
- use typed observations before flattening,
- use `result` for invalid actions.

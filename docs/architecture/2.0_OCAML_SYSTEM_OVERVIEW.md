# OCaml System Overview

This document is a clean-room implementation guide for an OCaml rewrite of the game. It is more concrete than a high-level pitch, but it still avoids copying source code structure line-for-line from the current Python prototype.

The goal is to answer questions like:

- what data should exist,
- where it should live,
- what each function is responsible for,
- which storage choices are reasonable,
- and how the pieces fit together without accidentally tangling the engine with RL code.

## Design Philosophy

Use **modules + records + variants** rather than OCaml objects.

Why:

- records are easy to inspect in tests,
- variants make illegal states harder to express,
- modules keep responsibilities clear,
- immutable updates make replay and debugging easier,
- and the same design works well for both deterministic simulation and RL rollouts.

The engine should be mostly pure. If randomness is needed, thread it explicitly through the game state or through a small RNG wrapper.

## Top-Level Architecture

The cleanest split is:

1. `engine`
2. `env`
3. `training`
4. optional `ui`

### `engine`

Owns:

- rules,
- clock,
- player state,
- referee state,
- death/revival,
- history,
- and the authoritative match state machine.

It should know nothing about tensors, neural networks, PPO, web UI, or dialogue generation.

### `env`

Wraps the engine into a step-based interface.

Owns:

- action encoding,
- legal action masks,
- observation construction,
- reward calculation,
- choosing or querying the opponent,
- and episode reset/termination handling.

### `training`

Owns:

- network definitions,
- rollout storage,
- PPO or imitation training,
- checkpointing,
- evaluation,
- and experiment configs.

### `ui`

Owns:

- CLI or frontend rendering,
- player prompts,
- timeline display,
- flavor text,
- and presentation-only state.

Do not let the UI become the place where rule logic lives.

## Recommended Core Types

You do not need these exact names, but you do need these concepts.

## `config`

This should be the single source of truth for all tunable constants.

### What it must store

- turn durations,
- failed-check penalty,
- cylinder cap,
- leap-second window boundaries,
- within-round overhead,
- death-procedure overhead,
- survival-curve parameters,
- player physicality presets,
- optional rule switches.

### Recommended shape

Use one top-level record with nested records.

That is better than a giant flat record because it groups parameters by meaning.

Example grouping idea:

- `timing`
- `cylinder`
- `survival`
- `players`
- `rule_flags`

### Why nested storage is useful

- easier to inspect in tests,
- easier to override only one category,
- easier to serialize to JSON or TOML later,
- better documentation value.

### Alternative storage options

Option A: one flat record
- simplest to start,
- annoying once the parameter count grows.

Option B: nested record tree
- best default,
- slightly more verbose,
- best for a school project because it stays readable.

Option C: module constants only
- avoid this for the final version,
- too rigid for experiments and RL sweeps.

## `player_state`

Represents one player's persistent match state.

### What it must store

- stable identifier,
- display name if needed,
- physicality,
- current cylinder amount,
- total time dead,
- death count,
- life status,
- optional death history.

### Recommended shape

Use a record.

Important fields to think about:

- `id`
- `name`
- `physicality`
- `cylinder_seconds`
- `ttd_seconds`
- `deaths`
- `status`
- `death_history`

### Status storage options

Option A: `alive : bool`
- simplest,
- too weak once you want temporary dead / permanently dead / revived states.

Option B: variant, recommended
- `Alive`
- `Dead_temporary`
- `Dead_permanent`

This is better because the death sequence is an actual game concept.

### Death history storage options

Option A: `int list`
- easiest,
- enough if you only need past death durations.

Option B: richer record list
- each entry stores duration, round, half, survival probability, survived flag.
- better for replay analysis.

Recommended default:

- start with a record list if you already know you want post-game analysis,
- otherwise start with `int list` and upgrade later.

### What should not be stored here

- current role,
- current game clock,
- opponent beliefs,
- reward,
- RNG.

Those belong elsewhere.

## `referee_state`

Represents global CPR / revival state.

### Minimum fields

- CPR count,
- optional fatigue scalar or derived stats,
- optional explicit health / condition later if you model Yakou more deeply.

### Storage options

Option A: just `cprs_performed : int`
- enough for the first engine,
- derive everything else from it.

Option B: richer record, recommended if you want canon-like behavior later
- `cprs_performed`
- `fatigue_level`
- `effective_quality`
- `stress_state`

Even if only `cprs_performed` matters today, a record makes later expansion painless.

## `game_state`

This is the master state for one match.

### It must store

- config reference or passed config externally,
- both players,
- referee,
- absolute game clock,
- current round,
- current half,
- first dropper or start parity,
- terminal state,
- winner/loser,
- history,
- RNG state if the engine owns randomness.

### Player storage options

Option A: named fields
- `player_a`
- `player_b`

Good when the game is always exactly two players. This is my recommendation.

Option B: map keyed by `player_id`
- more generic,
- slightly heavier,
- useful only if you expect generalized multi-agent tooling.

### Role/parity storage options

Option A: store current dropper directly
- easiest to reason about.

Option B: store `first_dropper` plus `current_half`
- derive current roles from parity,
- slightly more elegant if your role swaps are fully regular.

Recommended default:

- store `first_dropper`, `round_index`, and `current_half`,
- derive current roles with one helper.

### Terminal storage options

Option A: `game_over : bool` plus optional winner
- common and simple.

Option B: dedicated match-status variant, recommended
- `In_progress`
- `Finished of winner_id * loser_id * finish_reason`

The variant is cleaner because it encodes more meaning and avoids half-filled terminal fields.

### History storage options

Option A: append-only list
- idiomatic and simple.

Option B: array buffer
- better for very high-throughput simulation,
- more annoying to update immutably.

Recommended default:

- use a list in the engine,
- convert to arrays only in rollout or analytics code if needed.

## `half_round_record`

This is your audit log entry for one resolved half-turn.

### It should include

- round index,
- half index,
- game clock at start,
- dropper id,
- checker id,
- drop time,
- check time,
- turn duration,
- whether the turn was in the leap-second window,
- ST gained,
- cylinder before and after if useful,
- death duration if any,
- survival probability if any,
- sampled survival outcome if any,
- result classification.

### Why this matters

This is the easiest way to debug everything:

- rules,
- route timing,
- strange AI behavior,
- RL rollout surprises,
- and later post-game explanations.

If you make this too sparse now, you will regret it later.

## Recommended Shared Variants

Put these in a small shared domain module.

Useful variants include:

- `player_id`
- `role = Dropper | Checker`
- `half = First | Second`
- `turn_outcome`
- `life_status`
- `finish_reason`
- `match_status`
- `validation_error`

Variants help prevent stringly-typed logic from leaking everywhere.

## Data Ownership Rules

These rules keep the architecture sane.

### `Player` owns

- local per-player updates,
- cylinder accumulation,
- death bookkeeping,
- revival reset.

### `Clock` owns

- leap-second detection,
- time formatting,
- round snapping,
- absolute clock advancement.

### `Rules` owns

- validating action legality,
- comparing drop vs check,
- computing ST,
- deciding whether injection triggers.

### `Survival` owns

- survival probability math,
- referee fatigue modifier,
- random revival roll.

### `Game` owns

- role assignment,
- calling the other modules in order,
- updating full match state,
- terminal resolution,
- writing history records.

### `Env` owns

- how engine state becomes observations and rewards.

If you keep these boundaries strict, your later RL work gets much easier.

## Main Function Responsibilities

These are the core responsibilities, not exact code signatures.

## `Clock` functions

### `in_leap_window`

Input:

- config,
- current absolute game clock.

Output:

- whether a half-turn starting now receives the leap-second duration.

Responsibility:

- only detect the window,
- no role logic,
- no action legality.

### `turn_duration_for_start_time`

Input:

- config,
- start time.

Output:

- normal turn duration or leap turn duration.

Responsibility:

- centralize duration calculation so the rule is not duplicated.

### `advance`

Input:

- current clock,
- elapsed seconds.

Output:

- new absolute clock.

Responsibility:

- plain addition only.

### `snap_to_next_round_boundary`

Input:

- config,
- current clock.

Output:

- next round-start wall-clock time.

Responsibility:

- handle the leap second correctly,
- preserve wall-clock minute semantics.

### `format_wall_time`

Input:

- absolute game clock.

Output:

- human-readable time.

Responsibility:

- display only,
- but still mathematically correct across the leap second.

## `Player` functions

### `add_to_cylinder`

Input:

- player state,
- amount to add.

Output:

- updated player state,
- whether threshold was reached or exceeded,
- optionally the new cylinder amount.

Responsibility:

- no death resolution,
- just update cylinder and report overflow.

### `record_death`

Input:

- player state,
- death duration,
- optional metadata.

Output:

- updated player state.

Responsibility:

- increment death count,
- add to TTD,
- append history,
- mark temporarily dead.

### `revive`

Input:

- player state.

Output:

- updated revived player state.

Responsibility:

- set alive state,
- reset cylinder,
- keep historical damage intact.

### `mark_permanent_death`

Responsibility:

- move player into terminal dead state,
- do not reset anything.

### `safe_checks_remaining`

Input:

- config,
- player state.

Output:

- how many worst-case safe checks the player can absorb before fatal overflow.

Responsibility:

- be explicit about whether threshold equality still counts as safe.

This function must encode the exact interpretation you want and document it clearly.

## `Survival` functions

### `base_curve`

Input:

- config,
- current death duration.

Output:

- baseline survival chance before modifiers.

### `cardiac_modifier`

Input:

- config,
- prior total time dead.

Output:

- multiplier for accumulated damage.

### `referee_modifier`

Input:

- config,
- CPR count or referee state.

Output:

- multiplier for CPR fatigue.

### `survival_probability`

Input:

- config,
- referee state,
- player state before this death is added,
- current death duration.

Output:

- final clamped survival probability.

Responsibility:

- combine all factors,
- handle hard fatal threshold,
- avoid double-counting current death into prior TTD.

### `attempt_revival`

Input:

- RNG state,
- probability.

Output:

- survived or not,
- updated RNG state.

Responsibility:

- only sample,
- do not mutate player or game state here.

## `Rules` functions

### `validate_drop_time`

Input:

- turn duration,
- proposed drop time.

Output:

- success or validation error.

Responsibility:

- enforce `1 <= drop_time <= turn_duration`.

### `validate_check_time`

Input:

- config,
- proposed check time.

Output:

- success or validation error.

Responsibility:

- enforce checker action limits according to your chosen rule interpretation.

### `check_succeeds`

Input:

- drop time,
- check time.

Output:

- boolean success/failure.

Responsibility:

- encode the game's core timing relation in one obvious place.

### `squandered_time`

Input:

- drop time,
- check time.

Output:

- ST value.

Responsibility:

- probably only valid when the check succeeded.

### `resolve_half_outcome`

Input:

- config,
- checker state,
- drop time,
- check time,
- turn duration if needed.

Output:

- a rich local-outcome record.

### What `local_outcome` should store

- `check_success : bool`
- `st_gained : int`
- `penalty_added : int`
- `updated_checker`
- `injection_triggered : bool`
- `death_duration : int option`
- `result_class`

Responsibility:

- do local resolution only,
- do not roll revival,
- do not decide the winner,
- do not advance the clock.

## `Game` functions

### `init`

Input:

- config,
- players,
- first-dropper rule or explicit assignment,
- seed or RNG source.

Output:

- initial game state.

Responsibility:

- construct a valid match state,
- establish initial round/half/clock values.

### `roles_for_current_half`

Input:

- game state.

Output:

- dropper and checker ids or records.

Responsibility:

- centralize role derivation.

### `step_half_round`

Input:

- current game state,
- drop time,
- check time.

Output:

- updated game state,
- the half-round record,
- or a result type that contains both.

Responsibility:

- validate actions,
- compute turn duration,
- resolve local outcome,
- run death/revival if needed,
- advance clock,
- update history,
- move to next half/round,
- update terminal status if needed.

This is the main engine function.

### `step_round`

Useful convenience wrapper for playing two halves in sequence.

### `state_summary`

Return a simplified read model for CLI/UI/tests.

Good summary fields:

- round,
- half,
- current wall-clock display,
- leap-turn flag,
- both cylinders,
- both TTD values,
- both death counts,
- referee CPR count,
- match status.

## How To Store Time

Use integer seconds everywhere in the engine.

Avoid floats unless you have a very specific reason.

Why:

- the rules are second-based,
- comparisons are exact,
- tests become cleaner,
- leap-second behavior is easier to reason about,
- and you avoid weird formatting errors.

If you later want probabilities or normalized features, convert to floats outside the core rule comparisons.

## How To Store History

Recommended default:

- append immutable `half_round_record`s to a list.

Also consider two layers of history:

1. core engine trace
2. analysis trace

The core trace stores rule events only.

The analysis trace can later add:

- AI beliefs,
- explanation tags,
- dialogue selected,
- confidence scores,
- route-variation labels.

That separation is useful because you do not want AI debugging concerns to pollute your pure engine records.

## Error Handling Strategy

Prefer result types over exceptions for most engine-facing validation.

Examples:

- illegal action,
- stepping after terminal state,
- inconsistent role request,
- impossible state.

Reasonable pattern:

- use `result` for expected user/agent mistakes,
- reserve exceptions for true internal invariants being broken.

## Determinism Strategy

You want replayable matches.

Recommended options:

### Option A: store RNG state inside `game_state`

Pros:

- one self-contained match object,
- replay is easy.

Cons:

- slightly less pure-looking.

### Option B: pass RNG separately through every stochastic function

Pros:

- very explicit.

Cons:

- more plumbing.

Recommendation:

- for a school project, storing RNG state in `game_state` is perfectly reasonable.

## Suggested Build Order

1. Freeze shared types and config.
2. Implement the clock and leap-second model.
3. Implement player and referee state updates.
4. Implement survival math.
5. Implement half-turn resolution.
6. Implement the full game orchestrator.
7. Build scenario tests from the manga timeline.
8. Add observation and reward layers.
9. Add scripted opponents.
10. Add RL training code.

## What To Decide Early

These are the choices most likely to affect everything later.

1. whether checker actions are always capped at 60 or can expand to 61 if deduced,
2. whether the engine stores RNG or receives it externally,
3. whether history is minimal or analysis-rich,
4. whether referee fatigue is only CPR-count based or a richer state,
5. whether the game state exposes players as named fields or via a map.

If you want the shortest path with the least pain later, I would pick:

- checker capped at 60 by default,
- RNG stored in `game_state`,
- analysis-rich history,
- referee as a record with room to grow,
- players as named fields in a two-player game state.
